# 關鍵設計決策與取捨原則 (Design Philosophy)

> 本文件記錄了 Zenku 在架構設計過程中的核心決策，以及在彈性、穩定性與安全性之間所做的取捨。

---

## 1. 介面生成：元件組合 vs. 程式碼生成
**決策：採用「資料驅動的元件組合 (Building Blocks)」，而非直接生成原始程式碼。**

*   **做法**：預定義一組高度抽象化的 UI 元件（如 `TableView`, `FormView`, `DashboardWidget`），由 AI (UI Agent) 挑選並組合成 JSON 定義。
*   **取捨原因**：
    *   **穩定性**：避免 LLM 生成不合法的語法或存在安全漏洞（如 XSS）的程式碼。
    *   **可維護性**：當核心 UI 元件升級時，所有的 View 會自動套用新功能，無需重新生成。
    *   **預測性**：確保介面風格的一致性，不會因為模型隨機性導致 UI 風格突變。

## 2. 通訊架構：中心化調度 vs. 去中心化協作
**決策：採用「Orchestrator 中央集權模式」，初期不使用 Event Bus。**

*   **做法**：所有 Agent 之間禁止直接通訊，所有訊息必須流經 Orchestrator。
*   **取捨原因**：
    *   **低複雜度**：易於追蹤對話流與偵錯，減少因 Agent 間無限循環觸發導致的 Token 浪費。
    *   **全局掌握**：Orchestrator 能隨時存取當前所有 Agent 的狀態，做出更精準的意圖路由。
    *   **未來擴展**：當系統成熟後，此架構可輕易演進為基於事件驅動的非同步協作。

## 3. 安全防護：Context 隔離與最小權限
**決策：對 Agent 實施嚴格的上下文隔離與權限管控。**

*   **做法**：
    *   **權限隔離**：例如 `Query Agent` 僅擁有 `SELECT` 權限；`UI Agent` 無法存取真實業務資料。
    *   **Context 隔離**：每個 Agent 只會收到與其任務相關的系統元資料（Metadata），不會看到無關的定義。
*   **取捨原因**：
    *   **安全性**：防止 AI 意外執行越權操作（例如在查詢資料時誤刪資料表）。
    *   **Token 效率**：大幅縮減傳送給模型的上下文長度，降低成本並提升回應速度。

## 4. 容錯設計：設計日誌 (Design Journal) 與時光機 (Undo)
**決策：將系統變更視為「可逆的交易流」。**

*   **做法**：每一筆 Schema 或 UI 的變更都會記錄原始需求、變更原因及反向 SQL/Diff。
*   **取捨原因**：
    *   **AI 幻覺保護**：當 AI 做出錯誤決定時，使用者可以透過「Undo」指令瞬間還原，降低開發挫折感。
    *   **跨 Session 記憶**：Journal 成為 AI 了解「為什麼系統會變成現在這樣」的重要參考資料。

## 5. 資料模型：確定性 Schema vs. 靈活 JSON
**決策：堅持使用確定性的資料庫 Schema (Schema-on-Write)。**

*   **做法**：必須先定義好資料表與欄位型態，才能開始存取資料，而非將所有資料塞入一個 JSON 欄位。
*   **取捨原因**：
    *   **資料完整性**：能利用資料庫原生的類型檢查與關聯約束（FK）。
    *   **效能**：確保在資料量增大時，索引與聚合查詢依然高效。
    *   **外部整合**：結構化的資料表更容易與 n8n、BI 工具或其他外部系統對接。
